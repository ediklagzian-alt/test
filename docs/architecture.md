## Слои и роли

- `domain` - сущности и правила предметной области (`Client`, `Account`, `Bank`)
- `factories` - создание нужного типа счета через единый вход (`AccountFactory`)
- `builders` - последовательная сборка клиента с обязательными/необязательными полями (`ClientBuilder`)
- `commands` - оформление операций в отдельные команды с поддержкой `undo`
- `main.py` - консольный сценарий для ручной проверки

## Основные сущности

### Client

- обязательные поля: имя, фамилия
- необязательные: адрес, паспорт
- клиент считается верифицированным только при наличии адреса и паспорта
- верификация влияет на лимиты операций

### Account (абстракция)

Общий контракт:

- `deposit(amount)`
- `withdraw(amount)`
- `transfer(target, amount)`

Реализации:

- `DebitAccount` - нельзя уходить в минус
- `DepositAccount` - нельзя снимать до конца срока депозита
- `CreditAccount` - можно уходить в минус в пределах кредитного лимита, при отрицательном балансе применяется комиссия

### Bank

Отвечает за:

- реестр клиентов и счетов
- выполнение транзакций
- хранение истории операций
- rollback (отмена последней или заданной транзакции)
- контроль лимита для сомнительных клиентов

## Бизнес-правила

1. Сумма операции должна быть больше нуля.
2. Дебетовый и депозитный счет не могут уходить в минус.
3. Депозитный счет запрещает снятие до истечения срока.
4. Кредитный счет не может выйти за пределы `-credit_limit`.
5. Для сомнительных клиентов операции снятия/перевода ограничены лимитом банка.

## Паттерны и обоснование

### Builder

Нужен для поэтапного создания клиента: обязательные атрибуты проверяются при `build()`, необязательные можно добавлять по мере появления данных.

### Factory Method

Позволяет централизованно создавать разные типы счетов и не размазывать логику конструктора по клиентскому коду.

### Command

Операция представлена объектом с единым интерфейсом `execute/undo`. Это дает:

- общую точку выполнения транзакций
- историю операций
- возможность отката
